Как проверить результат в терминале?

Запустите проект с помощью sbt run. Обычный curl не умеет работать с веб-сокетами, поэтому для проверки потока используйте
утилиту wscat (устанавливается через npm install -g wscat) или любой онлайн-клиент веб-сокетов:
# wscat -c ws://localhost:8080/api/ws/ticks

Вы увидите мгновенный реактивный результат (каждые полсекунды прилетает новая строка):
Connected (press CTRL+C to quit)
< {"symbol":"BTC","price":60012.45,"timestamp":"2026-07-19T02:00:01.123Z"}
< {"symbol":"BTC","price":59987.10,"timestamp":"2026-07-19T02:00:01.624Z"}
< {"symbol":"BTC","price":60034.88,"timestamp":"2026-07-19T02:00:02.125Z"}

Этот поток работает на Виртуальных потоках (Fibers). Если к вашей системе одновременно подключатся 50,000 трейдеров
с React-фронтенда, Scala-бэкенд не создаст 50,000 тяжелых системных тредов. Он будет асинхронно мультиплексировать
этот FS2-стрим, рассылая JSON-пакеты с минимальным потреблением оперативной памяти CPU.

Почему вы выбрали Cats Effect, а не Akka/Actors?

Для симулятора торговой платформы в рамках одного сервиса стек Typelevel (Cats Effect, FS2) дает меньший оверхед,
лучшую типизацию и гарантирует типобезопасную работу с асинхронными ресурсами. Акторы Akka были бы оправданы, если бы
мне требовалось распределить состояние Order Book между несколькими физическими серверами (Cluster Sharding) или
строить сложную отказоустойчивую иерархию супервизоров.

Для нашего симулятора, который крутится внутри одного микросервиса, тащить огромный кластерный фреймворк вроде Akka — это стрельба из пушки по воробьям (оверхед).

Связка Cats Effect + FS2 — это как если бы мы взяли Java 21 с Виртуальными Потоками (Loom), добавили туда реактивные
стримы (как RxJava / Project Reactor), завернули работу с ресурсами в идеальный try-with-resources и управляли состоянием
через быстрые AtomicReference. Это дает максимальную скорость и легкость для одного сервиса.

В java есть ta4j - а что есть в scala такого для алгоритмической торговли ?

С одной стороны, из-за стопроцентной совместимости с Java вы можете напрямую использовать ta4j, просто подключив её в build.sbt.
С другой стороны, в Scala-сообществе и FinTech-фондах принято решать задачи алготорговли с помощью мощных
математических, статистических и потоковых (Streaming) библиотек.

1. Использование Java-библиотек (включая ta4j) напрямую

// Достаточно просто добавить в build.sbt:
libraryDependencies += "org.ta4j" % "ta4j-core" % "0.15"

// Использование в Scala 3 (код лаконичнее, чем в Java)
import org.ta4j.core.indicators.SMAIndicator
import org.ta4j.core.BaseBarSeries

val series = BaseBarSeriesBuilder().withName("BTC_USD").build()
val sma = SMAIndicator(series, 14) // Готово!

2. Чисто функциональный потоковый анализ (The Typelevel Way)

Для высоконагруженной торговли классический подход ta4j (где данные лежат в массивных объектах BarSeries в памяти)
часто не подходит. В финтехе котировки идут бесконечным потоком.

3. Библиотеки для Quant-анализа и Machine Learning

Для бэктестинга сложных стратегий, математического моделирования и машинного обучения в Scala используются тяжеловесные инструменты:
- Breeze: Это аналог библиотеки NumPy для Scala. Она предоставляет мощные средства для линейной алгебры, работы с матрицами,
оптимизации алгоритмов и генерации случайных чисел (критично для симуляций Монте-Карло в хедж-фондах).
- Smile (Statistical Machine Intelligence and Learning Engine): Топовая JVM-библиотека для машинного обучения, у которой
есть великолепный родной Scala API (DSL). Она работает быстрее, чем аналоги на Python, и используется для предсказания
движения цен на основе паттернов.
- Spark TimeSeries (spark-timeseries): Если вам нужно провести бэктестинг торговой стратегии на терабайтах исторических
данных за 10 лет, обычный ta4j зависнет. В EPAM для этого используют Apache Spark со Scala-библиотеками для обработки
временных рядов (Time Series).
